# RustでVSTホストアプリをGPLv3ライセンスで出すためのライセンス監査表

> **前提**
>
> この表は、**Rust製のVSTホストアプリをGPLv3ライセンスで公開・配布する**という前提で、主要候補のライブラリ／SDKを整理したものです。  
> ASIO対応のため `cpal` の `asio` feature を有効化し、これに伴いプロジェクト全体をGPLv3とします。
> これは技術・実務上の整理であり、**法的助言ではありません**。最終判断の前に、実際の依存関係ツリー（Cargo.lock / third-party notices / SDK同梱物）を確認してください。

---

## 結論サマリ

- **ToneDockはGPLv3ライセンス**で公開します。ASIO対応を優先するため、`cpal`の`asio` featureを有効化します。
- **VST3 SDK自体は現在MITで使える**ため、ホストアプリでの利用に問題はありません。
- ASIO SDKはSteinbergのライセンス下にあり、`cpal`のASIO featureを有効化したバイナリの配布にはSteinberg ASIO SDKの利用許諾条件を満たす必要があります。
- **`vst3-sys` はGPLv3** なので、GPLv3プロジェクトではそのまま利用可能です。
- **`vst3-rs`、`cpal` は permissive 系** なので、GPLv3プロジェクトとの併用に問題はありません。

## 監査結果一覧

## 個別監査メモ

### 1. Steinberg VST 3 SDK
- 現在のVST 3 SDKは **MIT License**。
- SteinbergのFAQでは、**hostをGitHub等で公開してよい**、**ソース開示義務はない**、**バイナリ配布や販売も可能**とされています。
- したがって、**VST3ホストアプリをMITライセンスで出すこと自体は可能**です。

**監査コメント**  
VST3そのもののライセンス障壁は、以前よりかなり低くなっています。  
いまは「VST3 SDKが問題か」ではなく、**その周辺のRustクレートのライセンスが問題か**を見るべきです。

**判定:** 採用候補

---

### 2. RustAudio `vst3-sys`
- `vst3-sys` は **GPLv3**。
- README上でも、**GNU GPLv3** と明記されています。

**監査コメント**  
MITホストアプリを目指すなら、ここはかなり危険です。  
VST3 SDK自体がMITになっても、**使うbindingがGPLv3なら、そこで方針が崩れます**。

**判定:** 避け推奨

---

### 3. coupler-rs `vst3-rs`
- `vst3-rs` は **MIT / Apache-2.0** のデュアルライセンス。
- VST3 APIのRust bindingsとして提供されています。

**監査コメント**  
MITホスト方針との整合性は良いです。  
ただし、これは高レベルフレームワークではなく、**比較的低い抽象のbinding** なので、ホスト実装の大部分は自前になります。

**判定:** 有力候補

---

### 4. RustAudio `cpal`
- `cpal` は **Apache-2.0**。
- RustでクロスプラットフォームのオーディオI/Oを扱う低レベルライブラリです。
- Windowsでは **ASIO feature** の記載があります。

**監査コメント**  
MIT単独ではなくApache-2.0ですが、商用・OSSの両面でかなり使いやすいライセンスです。  
VSTホストそのものではないため、**これ単体では足りませんが、音声I/O基盤としては有力**です。

**判定:** 採用候補

---

### 5. `clack`
- `clack` は **MIT / Apache-2.0**。
- READMEでは、**CLAP audio plugins and hosts を作るための safe, low-level wrapper** と説明されています。

**監査コメント**  
もし将来的に **VST3 + CLAP の両対応ホスト** を狙うなら、かなり相性が良いです。  
VST3の置き換えではなく、**拡張路線**として見るのが正しいです。

**判定:** 併用有力

---

### 6. CLAP仕様 / `free-audio/clap`
- CLAPは **MIT License**。
- 公式系リポジトリや紹介ページでも、**オープンで自由なライセンス** が明示されています。
- 公式の参考ホスト `clap-host` も **MIT** です。

**監査コメント**  
「ライセンスの軽さ」という意味では非常に優秀です。  
今後、VST3だけに縛られず、**CLAPを先に強く対応する**という戦略も取りやすいです。

**判定:** 採用候補

---

### 7. NIH-plug
- NIH-plug本体は **ISC**。
- ただし、関連文脈では `vst3-sys` のGPLv3がたびたび論点になります。
- そもそも NIH-plug は **プラグインフレームワーク寄り** で、今回の主題である**ホストアプリ**の中心候補ではありません。

**監査コメント**  
「RustでVSTやる」文脈で有名ですが、**ホストアプリ用の本命**としては見なくてよいです。  
誤って情報を混ぜると判断を誤るので、監査表では補足扱いに留めるのが適切です。

**判定:** ホスト用途では主候補でない

## 実装方針（GPLv3構成）

### ToneDock 構成
- `vst3` + `cpal` (features = ["asio"]) + `eframe`/`egui`
- ライセンス: **GPL-3.0-only**
- ASIO低レイテンシ対応、WASAPI フォールバック

## 最終結論

**ToneDockはGPLv3ライセンスで公開します。**  
ASIO対応による低レイテンシの利便性を優先し、プロジェクト全体をGPLv3とします。

現在の監査では、次の整理が妥当です。

- **採用:** `vst3` (coupler-rs) / `cpal` (ASIO有効) / `clack`
- **GPLv3のため利用可能:** `vst3-sys`
- **注意:** ASIO SDKの同梱条件を満たすこと
